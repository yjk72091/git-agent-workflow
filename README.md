# Git Agent Workflow Skill

`git-agent-workflow` is a Codex skill for applying agent-aware Git discipline during AI-assisted coding work. It turns an article's Git practices for the AI era into a compact, reusable workflow that helps agents keep changes isolated, reviewable, reversible, and traceable.

## Source

This skill was designed from the article:

- Title: `万字干货｜AI 时代的 Git 版本管理，你用对了吗？`
- Publisher: `TRAE.ai`
- Published: `2026-04-28 17:40:38`
- Original link: <https://mp.weixin.qq.com/s/70hz6sYNwxErRkP7dkY8-Q>

The article's core argument is that agentic coding changes the assumptions behind traditional Git usage. Agents can modify many files autonomously, run in parallel, create noisy diffs, and leave little durable record of intent. The skill extracts the operational practices needed to make Git history useful under those conditions.

## What This Skill Covers

Use this skill when an agent needs to:

- Make code changes while preserving Git hygiene.
- Split a large agent-generated diff into atomic commits.
- Create checkpoint commits during long-running work.
- Write commit messages with agent context and Git trailers.
- Prepare or audit a PR generated by an AI coding agent.
- Avoid overwriting human WIP in a dirty working tree.
- Use `git worktree` to isolate concurrent agents.
- Draft Git rules for `AGENT.md`.
- Decide whether plain Git, `jj`, GitButler, or stacked PRs fit the workflow.
- Design a team process for AI-agent version control, review, and traceability.

The skill is not a general Git tutorial. It assumes the agent already knows standard Git commands and focuses on the extra discipline needed when AI agents participate in development.

## Human Constraints and Safety Boundaries

The skill intentionally keeps several actions under human control:

- Do not overwrite, revert, or stage uncommitted human work unless explicitly instructed.
- Do not work directly on `main` or `master` unless the repository policy or user says so.
- Do not force-push or rewrite published history without explicit approval.
- Do not merge PRs, delete branches, or change branch protection rules without explicit approval.
- Do not introduce `jj`, GitButler, gh-stack, hooks, or new CI guardrails into a repository without user agreement.
- Do not treat a clean text merge as proof that the code is semantically safe.
- Do not commit secrets, debug-only changes, unrelated formatting, or generated noise without review.

The agent should present cleanup plans before interactive rebase or history rewriting. For high-risk code paths such as auth, billing, data migrations, public APIs, or shared packages, it should surface risk and validation gaps rather than treating the Git operation as the whole task.

## Tool Calls and Operational Flow

The skill primarily guides tool use rather than bundling scripts. Expected tool calls include:

### Git State Inspection

```bash
git status --short --branch
git branch --show-current
git diff --stat
git diff
git diff --staged
git log --oneline main..HEAD
```

Use these before editing, before committing, and before final handoff.

### Branch and Worktree Isolation

```bash
git fetch origin
git switch -c agent/<topic> origin/main
git worktree add ../repo-task-<topic> -b agent/<topic> origin/main
git worktree list
git worktree remove ../repo-task-<topic>
```

Use worktrees when multiple agents or risky tasks should not share one working directory.

### Intentional Staging and Atomic Commits

```bash
git add -p
git commit
git commit --amend
git log --format='%(trailers:key=Agent-Task,valueonly)' main..HEAD
git log --grep='^Agent-Task:' --all
```

Final commits should follow Conventional Commits and include agent trailers such as `Agent-Task`, `Agent-Model`, `Agent-Decision`, and `Agent-Limitation`.

### History Cleanup

```bash
git rebase -i main
git diff --stat main...HEAD
```

Use this only when the branch history is safe to rewrite. The agent should propose the rebase plan first.

### GitHub or PR Tooling

Depending on the environment, the agent may use GitHub connectors, `gh`, or repository web UI flows to:

- Read repository and PR metadata.
- Inspect PR diffs and changed files.
- Create a PR after commits are pushed.
- Add PR context, validation notes, and limitations.

The skill does not assume GitHub Actions logs are available through a connector alone; CI log inspection may require `gh` or the web UI.

### Optional Modern VCS Tools

Use only when already adopted by the user or explicitly approved:

```bash
jj split
jj absorb
jj op undo
```

GitButler may be appropriate for virtual branches, hunk assignment, and stacked branch workflows, but should not be introduced silently.

## Repository Layout

```text
.
├── SKILL.md
├── agents/
│   └── openai.yaml
└── references/
    └── templates.md
```

- `SKILL.md`: the main skill instructions and trigger description.
- `references/templates.md`: reusable templates for commit messages, checkpoint commits, PR descriptions, `AGENT.md` rules, branch protection checklists, and history cleanup prompts.
- `agents/openai.yaml`: UI metadata for the skill.

## Typical Invocation

```text
Use $git-agent-workflow to prepare an agent-safe Git workflow for this coding task.
```

The skill should be invoked when the work involves Git-sensitive coding changes, PR preparation, reviewability, traceability, or multi-agent coordination.
